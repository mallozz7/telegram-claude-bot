Mobility + Vibe: a conversation with an AI to build a remote interface for Claude Code

Donato Mallozzi Claude Code CLI · Telegram · Python 2026

We are all using Claude Code or similar AI development assistants more and more. These are powerful tools: they understand code, read files, modify, refactor, propose.

In practice it's easy to spend entire hours watching them work on your behalf, perhaps while following online meetings, only needing to step in occasionally to provide a new prompt and assess the result.

But they have a hard boundary — they require you to be there, in front of the terminal.

One afternoon I was waiting for Claude to finish a fairly long analysis on a legacy codebase. I had to go out. No way to know when it would be done, no way to send a follow-up, no notification.

I closed the laptop and left, losing the thread.
The thought that I believe comes naturally to anyone in similar situations is that it's a real shame to have to be physically in front of the PC, with a terminal open.

That frustration became a question:

"I'd like to interact with Claude Code even when I'm not at my PC. Can we develop an interface that receives commands remotely, both text messages and audio?"

What seemed like a simple technical problem — exposing an interface — turned out to be much richer. In eight successive conversations with Claude, we built something I hadn't planned: a complete system, with persistent sessions, async tasks, voice transcription, real-time streaming and security controls. This article tells the story — the choices, the rethinks, and what I learned along the way.

A honest caveat first: open source projects already exist — some very popular, like OpenClaw — that probably tackle this problem and do much more, although their focus is more on the personal assistant than on the code assistant in vibe coding style. I looked at them. The point is that doing much more was not what I was looking for. I wanted something simple, that I understood piece by piece, that I could control, modify and shut down at any time. Building from scratch, in this case, was not reinventing the wheel — it was choosing a wheel of the right size.

The first choice: which channel?

Before writing a single line of code, there was an architectural decision to make: how to get the prompts to the PC? The obvious options were a REST API, a web app, or some messaging service.

We chose Telegram, and not for trendy reasons. The concrete technical reason is that Telegram uses long polling — the bot calls Telegram, not the other way around. This means zero network configuration: no port forwarding, no open ports, works behind any corporate firewall. For a tool running on my home PC, it was the difference between "always works" and "depends on the router settings".

On top of that, Telegram natively handles voice messages — it compresses them in OGG/Opus and makes them downloadable via API. That feature, which at first seemed like a bonus, turned out to be central in many of the subsequent phases.

The initial setup was almost embarrassingly simple: create a bot on @BotFather, get a token, and the PC starts receiving messages.

Phase 1 — A prototype in a few files

The conversation
👤
I'd like to interact with Claude Code even when I'm not at my PC, both with text and voice messages.
🤖
Telegram is perfect for a prototype: long polling (no port forwarding), native voice messages, excellent mobile app. Setup in 10 minutes.

The first prototype was deliberately minimal. Three Python files, one main dependency (python-telegram-bot), and a clear idea of what it had to do: receive a message, pass it to claude --print, and reply with the output.

The biggest risk in this kind of project is wanting to do everything at once. I resisted. The prototype did exactly one thing: turn a Telegram message into a Claude Code prompt. No session management, no background tasks, no settings. Just the basic loop.

One choice made immediately, though, was non-negotiable: the user whitelist. Claude Code has full access to my PC's filesystem. A Telegram bot without access control is an open door to anyone who knows the bot's name. The Telegram user ID is assigned by Telegram's servers and cannot be spoofed client-side — a simple but solid barrier.

Voice worked via Whisper: the bot downloaded the audio file, transcribed it, showed the transcription in chat, then sent the text to Claude. That "show the transcription" was a small but important UX detail: the user immediately understands what the bot heard, and can correct it if Whisper misunderstood.

At the end of the first hour I had something that actually worked. I could send a message from my smartphone and receive Claude Code's response. The satisfaction of that first end-to-end test is hard to describe.

And at that point you can already imagine: I activated the bot service on my PC and left, continuing to improve the bot remotely. With each new feature, Claude CLI updated the code on my PC, ran the tests and restarted the service to make the changes available immediately.

Phase 2 — The session problem

A fundamental aspect became immediately clear: being able to reconnect to a previously opened session and continue it leveraging its context was essential.

The conversation
👤
The list command shows available sessions (id and title) and the resume command lets me reactivate any of them, even by title.
🤖
I'll add session_manager.py that reads the JSONL files from ~/.claude/projects/ and extracts ID, title and open status via /proc.

Using it for a few hours, I hit the next limit. Claude Code saves every conversation as a JSONL file on disk, and --continue resumes the last session. But I work on different projects — and I often want to resume a specific conversation, not necessarily the last one.

The question I asked Claude was very precise: I want to be able to list sessions and resume one by title or partial ID. The answer required understanding how Claude Code organises data on disk — an undocumented directory structure — and building a reader that extracted useful information from it.

The most interesting part of this phase was detecting currently open sessions. Claude Code leaves no lock files. The only way to know if a session is open is to look at running processes — on Linux, by inspecting /proc/*/cmdline. If the cmdline contains --resume <uuid>, the match is exact and certain. If not, the most recently modified session is marked as "probably open" — transparency about uncertainty rather than pretending a certainty that doesn't exist.

Phase 3 — The async turning point

At this point, why limit ourselves to synchronous requests: with the task command I want to launch a long computation and continue with other interactive sessions.

The conversation
👤
I need progress tracking — I want to be notified as soon as Claude finishes a long-running task in a current session.
🤖
Architectural change: subprocess.run() → asyncio.create_subprocess_exec(). The bot stays responsive and sends a push notification when done.

This was the most significant rethink of the entire project.

The original prototype used subprocess.run() — blocking by definition. While Claude was processing, the bot was silent: it couldn't reply to other messages, couldn't receive a /cancel, couldn't do anything. For short responses it's not noticeable. For tasks that take minutes, it's a serious problem.

Migrating to asyncio.create_subprocess_exec() changed the nature of the system. The bot became genuinely async: you can start a long task, keep using the bot normally, and receive a push notification when Claude has finished — with the complete output in the message. You can also start multiple tasks in parallel, monitor them, and cancel a specific one.

But it also needs to be possible to interrupt a long-running task if you realise you've missed important information in the prompt. Cancellation was a delicate detail: terminating both the OS process (proc.kill()) and the asyncio task (asyncio_task.cancel()), and handling error propagation so the user receives a confirmation notification even when cancelled.

Looking back, this is the phase that transformed the bot from "useful widget" to "serious tool".

Phase 4 — Giving voice to commands

The conversation
👤
I need a way to invoke /.. commands in voice messages, not just as free prompts.
🤖
I'll use Claude Haiku as a classifier: transcribe → structured JSON → dispatch to the same handler as the text command. ~0.5 seconds of latency.

Voice messages worked well as free prompts — you say something, Claude responds. But I soon wanted to do more: say "show me the running tasks" and have the bot understand I was asking for a list, not a prompt to pass to Claude Code.

The first impulse was to build a keyword matching system. "Task" → /tasks, "cancel" → /cancel. Easy to implement, impossible to maintain, fragile by definition.

The most elegant solution was to use an LLM to classify the intent — specifically Claude Haiku, the fastest and most economical model. A single API call, a system prompt describing the available commands, and the model responds with structured JSON indicating which command to run and with what arguments. About half a second of additional latency, zero maintenance.

What struck me about this choice is a subtle irony: I'm using an AI to interpret voice commands that are then passed to another AI. But it works — and works well even with ambiguous phrases and colloquial constructions.

An important design detail: if the Haiku call fails for any reason (rate limit, network error), the bot doesn't block. The transcription is passed directly to Claude Code as a free prompt. Complete silence from the user's perspective, service continuity.

Phase 5 — The silence during the wait

So far so good, but everything felt too detached — I had no feedback from my assistant during its processing.

The conversation
👤
Earlier you suggested JSON streaming — what exactly do you mean?
🤖
claude --output-format stream-json emits JSONL events in real time. I edit the Telegram message every N seconds with the partial text + cursor ▌.

There's a tension in the design of any waiting system: the user doesn't know whether something is happening or whether something has broken. A "⏳ Claude is processing…" frozen for two minutes creates anxiety.

Claude Code supports a --output-format stream-json flag that emits JSONL events as it processes, instead of waiting for completion. Telegram doesn't support character-by-character streaming, but it allows editing an already-sent message. The solution was to combine the two: accumulate partial text from JSON events, and update the Telegram message every two seconds with the current text and a ▌ cursor.

The throttling was necessary to avoid exceeding Telegram's rate limits (about 20 edits per minute per message), but it also proved to be a UX improvement: updates every two seconds are readable, every 200ms would be disorienting.

A small detail I liked: the ▌ cursor at the end of the partial text. It's not technically informative — it only says "still writing" — but it transforms the wait from anxiety to anticipation.

Technical note Background tasks (/task) deliberately use non-streaming mode. You're waiting for the final notification anyway — you're not watching the screen in real time.

Phases 6 + 7 — Attachments and settings: the details that make the difference

At this point I started needing to see some files modified or produced by the bot — I needed a way to open them on my smartphone.

The conversation
👤
A /sendme command for receiving files as attachments would be useful. And I want to set the current directory and streaming frequency before launching commands.
🤖
Two additions: /sendme with automatic ZIP for multiple files + declarative SETTINGS_SCHEMA with /set and /settings. One new parameter = one line in the dictionary.

These two phases arose from practical needs that emerged from use.

The first: I often asked Claude to generate files, reports, or scripts — and then had no easy way to retrieve them from my smartphone. The attachment command turned out to be richer than expected: a single file arrives as a direct attachment, but if a glob pattern is used (src/*.py) the bot creates a ZIP in memory — no temporary files on disk — and sends it as an archive. Automatic name sanitisation was a practical detail: src/*.py becomes src_py.zip, special characters included.

The second: I wanted to configure the bot's behaviour without having to edit the .env file and restart the process. Workdir, streaming frequency, timeout, transcription language. The design choice here was to use a declarative schema: adding a new parameter requires a single entry in a dictionary, and all validation logic, panel rendering and default reset is handled automatically. This kind of decision — investing in an abstraction instead of copy-pasting — pays off by the second modification.

Phase 8 — The security you don't see

The conversation
👤
Let's add a security check to the transcription: if it contains dangerous terms, the bot asks for confirmation before passing it to Claude.
🤖
Regex patterns on 6 risk categories. If triggered, I show Telegram inline buttons ✅/❌. The payload is stored in user_data until tapped.

The last problem was subtle. Text messages I type consciously — hard to accidentally say "delete everything". Voice messages are different: Whisper might transcribe unexpectedly, or I might speak in a noisy context with unpredictable results. With Claude Code having access to the filesystem, a transcription error on a destructive command could cause real damage.

The solution was a filter on risky pattern categories — file deletion, privileged commands, unexpected network operations, arbitrary shell execution, credential access — and an explicit confirmation request via Telegram inline buttons when one of these patterns is detected.

The design aspect that convinced me is that the confirmation arrives before execution, with the transcription visible. The user sees exactly what the bot understood and decides whether to proceed. It's not an obstacle — it's transparency.

The payload (transcription + already-classified intent) is temporarily stored, and the button retrieves it on tap. No reprocessing, no additional API calls.

What I learned

Looking back at the eight phases, some things seem clearer now than they were at the start.

Starting with something that works is more important than starting with something complete. The first day's prototype was almost ridiculously simple. But it worked, and that direct experience guided every subsequent decision in a way that no upfront planning could have.

Channels matter as much as content. The choice of Telegram was not aesthetic — it solved three problems at once: authentication, transport without network configuration, and native audio support. Choosing the wrong channel would have made everything else much harder.

The best features grow from real needs — or real frustrations 🙂 Real-time streaming grew from the embarrassing silence during the wait. Sessions grew from confusion between different projects. The safety checker grew from the awareness that voice is a less controlled input than text. None of these needs were obvious before using the system.

Building with an AI to control an AI has something circular and productive about it. All the code was developed in conversation with Claude Code, which was running in the same session we were extending. There's a feedback loop in this that speeds up decisions and forces you to be precise about what you want.

In summary

In eight conversations started from a single frustration, we built something I hadn't planned to build. The result is a system that takes a terminal tool and makes it accessible everywhere — with text, voice, attachments, persistent sessions, real-time updates and a reasonable level of security.

The most interesting thing is not the final system, though. It's the process: how each iteration revealed the next problem, how architectural choices made for practical reasons turned out to be right in retrospect, and how collaboration with an AI agent — in the same session where we were building the tool — changed the way I thought about the problem.

From that point on it was hard to stop. That evening I kept going from the couch. Now my assistant also has:
